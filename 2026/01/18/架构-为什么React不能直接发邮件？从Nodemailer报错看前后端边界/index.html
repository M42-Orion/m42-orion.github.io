Module not found: Error: Can't resolve 'net' in 'node_modules/nodemailer/lib/smtp-connection'
Module not found: Error: Can't resolve 'tls' ...
Module not found: Error: Can't resolve 'fs' ...

这不仅是技术报错，更是一次架构层面的“降维打击”。今天我们就借这个案例，深入聊聊为什么有些事前端注定做不了，以及 Node.js 在前后端分离中的真正角色。

一、 技术之墙：浏览器是个“沙盒”

首先解释为什么会报错。这需要从网络协议的底层说起。

你写的 React 代码虽然是在 Node.js 环境下编译（通过 Webpack 或 Vite），但最终生成的 JS 文件是运行在用户的浏览器里的。

1. 协议的生殖隔离：HTTP vs SMTP

  • Web 世界的通用语：浏览器天生是为 HTTP/HTTPS 协议设计的。当你调用 fetch 或 axios 时，浏览器底层封装好了 TCP 连接、握手、TLS 加密等一系列复杂操作，只暴露给你一个简单的请求接口。
  • 邮件世界的通用语：发送邮件使用的是 SMTP (Simple Mail Transfer Protocol) 协议。这是一个基于 TCP 长连接 的纯文本协议，需要客户端与服务器进行多次“对话”（握手 -> 验证身份 -> 发送发件人 -> 发送收件人 -> 发送内容 -> 挥手）。

2. 缺失的底层能力：Socket

  • Node.js 环境（后端）：拥有 net 模块，它允许代码直接创建原始的 TCP Socket。nodemailer 正是利用这个能力，手动构建 SMTP 协议的数据包，通过 TCP 管道直接发给邮件服务器（如 QQ 邮箱的 smtp.qq.com:465）。
  • 浏览器环境（前端）：为了防止恶意网页随意连接你内网的打印机或路由器，浏览器把 JS 关在一个“沙盒”里。它严厉禁止 JS 代码直接创建 TCP Socket。
    • “等等，WebSocket 呢？” —— WebSocket 虽然名字带 Socket，但它本质上还是基于 HTTP 握手升级上来的应用层协议，并不是原始的 TCP Socket。你无法用 WebSocket 去连接一个标准的 SMTP 服务器，因为对方听不懂 WebSocket 的握手包。

所以，当 nodemailer 试图在浏览器里调用 require('net') 去创建一个 Socket 时，Webpack 会告诉你：“对不起，浏览器里没有这个东西。”

3. 文件系统的隔离

很多 Node.js 库依赖 fs 模块读取本地的证书文件或附件。浏览器出于隐私保护，绝对不允许 JS 随意扫描用户的硬盘。这也是一道无法逾越的墙。

二、 安全之墙：源代码是“裸奔”的

如果说技术限制还可以通过“魔改库”来绕过（比如用 HTTP API 替代 SMTP），那么安全问题则是死穴。

前端代码没有秘密

不管你用 Webpack 怎么压缩混淆，前端代码最终都会被下载到用户的电脑上。
任何一个略懂技术的用户，只需要：

  1. 打开浏览器控制台 (F12)。
  2. 点击 Sources 面板。
  3. 搜索 “qq.com” 或 “pass”。

你的密码就赤裸裸地躺在那里。这意味着任何人都可以拿到你的授权码，用你的邮箱给全世界发送垃圾邮件。

结论：永远不要在前端代码里保存任何 Secret（密码、密钥、Token）。

三、 认清 Node.js 的两张面孔

这个案例暴露了一个常见的认知误区：混淆了“作为构建工具的 Node.js”和“作为服务器的 Node.js”。

场景 角色 运行位置 能力范围 例子
前端开发 构建工具 开发者电脑 / CI服务器 读写文件、编译代码 Webpack, Vite, Create-React-App
前端运行 脚本 用户浏览器 DOM操作、HTTP请求 React, Vue, jQuery
后端运行 服务器 服务器 TCP连接、数据库、文件系统 Express, Koa, NestJS

你在 Login.tsx 里写的代码，属于第二行。它虽然是用 Node.js 工具链编译出来的，但它不是 Node.js 程序，它只是运行在浏览器里的 JavaScript。

四、 正确的架构：后端作为“代理人”

既然前端做不了，那该怎么发邮件？
这正是后端（Backend）存在的意义。我们需要一个拥有特权的“中间人”。

解决方案

  1. **前端 (React)**：

    • 只负责收集用户输入（邮件内容）。
    • 通过 HTTP POST 请求把数据发给后端。
    • 不需要 知道邮箱密码。

  2. **后端 (Node.js/Python/Go)**：

    • 运行在服务器上，环境受控且安全。
    • 持有密码（从环境变量读取，不进版本控制）。
    • 引入 nodemailer，通过 TCP 连接邮件服务器发送邮件。

五、 总结

这次“发邮件失败”的经历，实际上是一堂生动的架构课。它告诉我们，前后端分离不仅仅是代码的分离，更是 信任边界 (Trust Boundary) 的划分。

  • 前端 (Client)：是不可信环境。它受限于浏览器沙盒，且代码完全公开。它只负责“展示”和“请求”。
  • 后端 (Server)：是可信环境。它拥有系统底层权限（网络、文件、数据库），且可以安全地保管机密。它负责“执行”和“守护”。

下次当你发现某个 Node.js 库在 React 里跑不通，或者需要在代码里写密码时，停下来想一想：“这个功能，是不是本该属于后端的职责？”
