登录成功立刻 401：Cookie Secure 陷阱

项目：SGCC 智能服务控制台 · 学习记录

现象

部署到服务器后第一次试登录，看到的是这样的体验：

  1. 输入邮箱 → 收到验证码 → 上传 auth.key → 提交
  2. 浏览器跳出登录页，地址栏短暂出现 /home
  3. 马上又被弹回 /login
  4. 手动改地址栏到 /work-order，页面能渲染出来
  5. 一点击列表里的任何东西 → 又被弹回 /login

后端日志里没有任何报错：

登录是成功的。但每次跳转都瞬间被踢出。

诡异之处：登录确实成功了，但用户像是「持有了一张看不见的票」——服务端认为他登过，浏览器表现得他没登过。

排查思路

第一步：看 nginx 日志

中间隔了不到 1 秒。登录拿到了 200，下一个请求就被拒。

第二步：定位「拒」在哪儿

/api/auth/check 的实现：

req.cookies 是空的——浏览器没把 cookie 带回来。

那为什么不带？两种可能：

  1. 服务端根本没下发 cookie（Set-Cookie 头丢了）
  2. 服务端下发了，但浏览器主动丢弃了

第三步：F12 看响应头

打开 DevTools → Network → 找到 /api/auth/login 的响应：

Set-Cookie 头是有的，且包含 Secure 标记。

那就是浏览器丢了。

根因：Secure 标记 + HTTP 连接 = 浏览器丢弃

浏览器规范说什么

RFC 6265bis 与现代浏览器（Chrome 80+ / Firefox 75+ / Safari 14+）的实现：

A cookie with the Secure attribute MUST only be sent over a secure (HTTPS) connection. If the response that sets the cookie was delivered over an insecure (HTTP) connection, the cookie MUST be rejected.

直白点说：

连接 Set-Cookie: ...; Secure 浏览器行为
HTTPS 接受 存下来
HTTP 接受 直接丢，不存

我的部署是 http://39.102.116.106:3000——纯 HTTP，没有 HTTPS。结果浏览器收到响应后默默丢弃了这个 cookie。下一次请求自然没 cookie 可带。

我代码里的 bug

frontend/server/index.js：

我把 secure 直接绑死到 NODE_ENV === 'production'。

逻辑上的假设：「生产环境一定有 HTTPS」。

现实：这个项目还没有域名，也没有 TLS 证书——生产是 HTTP 的。所以「生产 = 一定 HTTPS」这个假设在这个阶段不成立。

顺便讲讲 cookie 的几个 flag

这次踩坑后我把 cookie 的安全 flag 重新理了一遍：

HttpOnly

意思：浏览器里的 JS 看不见这个 cookie。

  • document.cookie 读不出
  • 任何 XSS 注入的 JS 也读不出

防御目标：XSS 偷 session。即使页面被 XSS，攻击者也拿不到 token。

我们这里必须开——cookie 里装的就是 session token。

Secure

意思：只有 HTTPS 下浏览器才接受 / 才往外发。

防御目标：中间人在公开 Wi-Fi 嗅探明文 HTTP 流量偷 cookie。

需要 HTTPS 当前提。没 HTTPS 就不能开——开了浏览器直接丢，登录就坏了（就是我刚踩的坑）。

SameSite

意思：跨站请求时浏览器还要不要带这个 cookie。

模式 跨站请求带 cookie? 用途
Strict 完全不带 CSRF 强防御，但用户从外站点进来要重新登录
Lax 顶层导航（点链接）时带，子资源 / fetch 不带 默认值，体验+安全平衡
None 全带 跨域单点登录之类，必须配 Secure

防御目标：CSRF。

我们用 Strict，是因为 admin 控制台没必要支持「从外部链接直接进来」这种场景，安全优先。

三个 flag 的最佳组合

部署形态 HttpOnly Secure SameSite
当前（HTTP，纯内部访问） ✅ true ❌ false Strict
上 HTTPS 后 ✅ true ✅ true Strict
跨域 SSO（不适用本项目） ✅ true ✅ true None

「HttpOnly + Secure + SameSite」是 cookie-based session 的安全基线。任何一个不开都有特定的攻击向量能利用。

  1. nginx 加 443 监听 + 证书（Let’s Encrypt 免费）

  2. 打开 Secure：

  3. 加 HSTS 响应头（告诉浏览器以后只走 HTTPS）：

    注意 HSTS 一旦生效就锁死了——浏览器在 max-age 期内会拒绝访问 HTTP 版本。先 max-age=300（5 分钟）测稳了再调大。

  4. 重启 api：

  5. 登出再登录 让浏览器拿到新的带 Secure 的 cookie。

收获

1. 「无报错」不等于「无问题」

后端日志一片祥和：login_success。但用户体验完全坏掉。这种问题的特征是两边各自看都正常，组合起来不工作：

  • 服务端：「我下发 cookie 了啊」
  • 客户端：「我没收到 cookie 啊」
  • 中间的浏览器规范：「Secure + HTTP，丢了」

排查这种问题，要去看双方对话的中间状态——HTTP 头部、网络面板、浏览器存储。

2. 把假设写成代码之前先验证

我把 secure: NODE_ENV === 'production' 当成「不言而喻的最佳实践」抄下来，没意识到这条预设了「生产 = HTTPS」。这个预设在项目早期就是不成立的。

写「智能默认值」时要问：默认值不成立的场景下，会发生什么？

我的代码 secure: IS_PRODUCTION 的不成立场景就是「生产部署但没 HTTPS」——结果是登录直接坏掉，没有任何提示。这是个糟糕的失败模式。

3. 给运维留可观测点

修复后我加了启动日志：

下次再排查任何 cookie 相关问题，第一行日志就能看到当前生效的配置。关键配置一定要在启动日志打出来——不然 debug 时只能去翻代码反推。

4. 渐进式安全收紧

正确做法不是「一上来就把所有 flag 都开到最严」，而是「按部署能力分阶段收紧」：

  • 阶段 1：HTTP 部署 → COOKIE_SECURE=false + HttpOnly + SameSite=Strict
  • 阶段 2：HTTPS 接入 → COOKIE_SECURE=true + 加 HSTS
  • 阶段 3：全站 HTTPS 稳定运行 → HSTS max-age 调大 + preload

每一步都对应一个明确的能力提升。开过头会自伤，开不够会被打。
